## 二、AI 发现的主要问题
基于 OWASP Top 10 视角，AI 发现了以下安全问题：

### 1. 失效的访问控制（高危）
部分接口未严格校验登录状态，例如：
- `/api/task/create`
- `/api/task/grab`
- `/api/task/finish`
- `/api/task/history`

问题原因：
- Service 层部分方法未检查当前用户
- JWT 解析失败时仅清空 `ThreadLocal`，没有直接拦截请求

可能导致：
- 未登录用户访问受保护接口
- 越权操作任务数据

### 2. 信息泄露（中危）
全局异常处理器中存在：

```java
ApiResponse.error(500, "Internal server error: " + ex.getMessage())
```

会将内部异常信息直接返回前端。

风险：
- 暴露数据库结构
- 暴露后端实现细节
- 增加攻击者利用漏洞的可能性

### 3. 输入校验不足（中危）
任务创建和搜索接口缺少输入长度和格式限制，例如：
- `title`
- `keyword`
- `deadline`

风险：
- 恶意构造超长输入
- 数据异常
- 潜在 SQL 模糊查询滥用

### 4. 并发与状态竞争问题（中危）
任务抢单逻辑存在竞态条件：

- `existsByTaskIdAndUserId()`
- `save()`

之间可能出现并发重复抢单。

风险：
- 重复抢任务
- 人数超限

### 5. ThreadLocal 鉴权设计风险（低危）
项目使用：

```java
ThreadLocal<Long>
```

存储当前用户。

风险：
- 测试环境容易遗漏登录态
- 多线程场景可能产生数据污染
- 不利于后续扩展 Spring Security

## 三、修复内容

### 1. 增加登录校验
在以下 Service 方法中增加：

- `JwtAuthenticationInterceptor.getCurrentUserId()` 判空

涉及方法：
- `createTask`
- `grabTask`
- `finishTask`
- `cancelTask`
- `myTaskHistory`

未登录时统一抛出：

```java
AuthenticationException("用户未登录")
```

### 2. 修复异常信息泄露
修改全局异常处理：

- 修复前：`"Internal server error: " + ex.getMessage()`
- 修复后：`"服务器内部错误"`

避免向客户端暴露内部异常信息。

### 3. 增加输入合法性校验
增加以下校验项：
- 分页参数校验
- `keyword` 长度限制
- `title` 长度限制
- `deadline` 格式校验

防止非法输入导致异常。

### 4. 修复任务抢单并发问题
采用：
- 数据库原子更新
- 唯一索引兜底
- rollback 回滚人数

修复了重复抢任务问题。

### 5. 修复空指针问题
增加判断：

```java
task.getPublisherId() != null
```

避免 `NullPointerException` 导致服务异常。

## 八、遇到的问题和解决
1. 问题

单元测试频繁出现：

Authentication 用户未登录
解决

原因是：

ThreadLocal

中的用户信息未在测试环境中设置。

通过：

AuthTestUtil.setCurrentUserId()

模拟登录态解决。

2. 问题

deadline 字段解析失败。

解决

原实现仅支持：

ISO_LOCAL_DATE_TIME

后续增加：

yyyy-MM-dd HH:mm
yyyy-MM-dd HH:mm:ss
yyyy-MM-dd

多格式兼容解析。

3. 问题

任务抢单出现并发问题。

解决

采用：

原子更新 SQL
唯一索引
回滚机制

保证数据一致性。

心得体会

在 Vibe Coding 场景下，开发效率通常会优先于代码规范和安全性，但随着项目规模扩大，安全问题会逐渐暴露。

本次实践中，我体会到：

AI 可以快速发现潜在安全问题，提高审查效率
但 AI 给出的修改方案仍需要人工理解和验证
安全设计应该尽早融入开发流程，而不是最后补救
单元测试对于验证安全修复非常重要
鉴权、异常处理、输入校验等基础安全措施必须统一规范

同时我也认识到：

“能运行”并不代表“安全可靠”。

未来开发中，需要在开发效率与系统安全之间做好平衡，尽量在保证功能实现的同时，提前考虑安全设计与可维护性。
