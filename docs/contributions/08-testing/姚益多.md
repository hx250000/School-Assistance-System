## 遇到的问题与解决方案

1. 问题：测试文件名包含非法字符（`/`、空格），导致编译报错
   - 解决：统一使用英文驼峰命名规范，修改为纯英文、无特殊字符的文件名（如 `MyTaskDetailFragmentTest.kt`），避免编译异常。
2. 问题：测试中重复定义数据模型（如 `PointRecord`、`ShopItem`），导致与项目真实模型类型不匹配，报错 “Argument type mismatch”
   - 解决：删除测试中自定义的数据模型，直接导入项目真实的 model 类（`com.example.campustask.model` 包下的对应类），确保类型一致。
3. 问题：测试中使用 `emptyList()` 未指定泛型，报错 “Cannot infer type for type parameter 'T'”
   - 解决：明确声明泛型类型（如 `emptyList<ShopItem>()`），帮助 Kotlin 自动推导类型，解决编译报错。
4. 问题：依赖冲突（Mockito 依赖找不到），导致测试无法运行
   - 解决：改用 JUnit4 原生测试方案，不混用第三方依赖，确保测试环境稳定、可运行。
5. 问题：测试用例过于简单（如使用 `assertTrue(true)`），不符合严格测试要求
   - 解决：重写测试用例，覆盖空数据、非法数据、边界值等场景，确保测试用例能真实查出代码缺陷，提升测试质量。

## 心得体会

本次软件测试实验，我围绕安卓前端的 4 个核心页面（任务详情、任务列表、积分记录、商城），完成了全面的单元测试工作。

通过实验，我掌握了安卓前端组件交互测试、Mock 数据模拟、边界与异常场景测试的方法，学会了如何排查测试中的编译报错（如泛型不匹配、类型冲突、文件名非法等），并能通过测试发现代码中的真实缺陷（如空数据无容错、积分/价格未做合法性校验、图片资源异常会崩溃等）。
